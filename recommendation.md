# Production Recommendations

The most impactful change for production would be replacing the in-memory inverted index with a persistent storage layer (e.g., Elasticsearch or PostgreSQL with full-text search). This eliminates the current memory ceiling and enables the index to survive process restarts without relying on JSON snapshots. The single-process crawler could then be scaled into multiple worker processes coordinated through a message queue (Redis Streams or Kafka), with the URL frontier and visited set backed by Redis for shared, atomic deduplication across workers. Per-domain rate limiting should be extended with robots.txt compliance.

On the search and serving side, the relevancy model should move from raw keyword frequency to TF-IDF or BM25 scoring, and the API should sit behind a load balancer with query-level caching for hot terms. The dashboard polling should be replaced with Server-Sent Events for lower-latency metric updates. Operationally, the system should be containerized (Docker) with Prometheus metrics and structured logging so that crawl throughput, error rates, and query latency (p50/p99) are observable from day one. These changes would bring the architecture from a working single-machine prototype to a horizontally scalable, production-grade system.
